# HandoffOS Implementation Plan

This document is the build order for the HandoffOS MVP. Each phase has a clear deliverable and validation gate.

## Phase 1: NitroStack Foundation

**Goal:** Create a clean NitroStack TypeScript project that can run in NitroStudio and deploy through NitroCloud later.

Implementation:

1. Use the NitroStack CLI to scaffold the official TypeScript starter project in this repository.
2. Set the server identity to `handoffos-server`.
3. Add `.nvmrc` targeting Node `20.18.1`.
4. Keep the NitroStack module, decorator, Zod, widget, and TypeScript configuration generated by the CLI.
5. Remove starter-product naming so all visible metadata says HandoffOS.
6. Keep the starter feature only as framework plumbing until Phase 4 replaces it with the workflow module.
7. Verify installation, type generation, and production build.

Acceptance gate:

```text
npm install succeeds
npm run dev starts the MCP and widget development servers
npm run build succeeds
NitroStudio can discover the starter MCP server and widget
```

## Phase 2: Domain Contracts and Seed Data

**Goal:** Define the workflow model and a repeatable Priya onboarding scenario.

Implementation:

1. Create types for workflow nodes, events, evidence, findings, audit entries, simulations, and action plans.
2. Create an injectable workflow-state store interface and in-memory implementation.
3. Seed Priya Nair's onboarding workflow and mock source events.
4. Seed the missing Laptop Allocation task as the root cause.
5. Add a deterministic demo clock so the joining date, SLA state, and completion estimates remain stable.

Acceptance gate:

```text
The seeded state contains all onboarding stations
Manager Approval and HR Verification are completed
Laptop Allocation is absent or blocked
Downstream nodes reflect their dependencies
```

## Phase 3: Rules, Evidence, Health, and Simulation

**Goal:** Compute all workflow facts without AI.

Implementation:

1. Implement rules R-001 through R-007.
2. Emit structured findings with evidence references and risk points.
3. Calculate workflow health from the documented risk formula.
4. Identify the root blocker and critical path.
5. Implement cloned-state simulation for a proposed node resolution.
6. Produce before/after workflow summaries without mutating live state.

Acceptance gate:

```text
Laptop Allocation fires Missing Dependency and Critical Path Blocked findings
Every finding links to evidence
Initial health is 62
Completing Laptop Allocation in simulation improves health to 86
Simulation leaves live state unchanged
```

## Phase 4: MCP Resources, Tools, and Prompts

**Goal:** Expose the workflow engine as a reusable NitroStack MCP server.

Implementation:

1. Create the workflow resource handlers for state, events, findings, audit log, and rules.
2. Add Zod-validated tool contracts for event ingestion, blocker detection, simulation, action planning, and action execution.
3. Add the two prompts: `explain_blocker` and `manager_summary`.
4. Ensure tool output includes structured workflow data as well as readable text.
5. Restrict execution to actions returned by a plan and require the approving actor.
6. Append a durable-in-memory audit entry for every state-changing action.

Acceptance gate:

```text
NitroStudio lists five resources, five tools, and two prompts
Resources return the expected seeded content
Tools reject invalid schemas
Unapproved actions are rejected
Approved actions recalculate workflow state
```

## Phase 5: Transit-Map Widget

**Goal:** Turn workflow output into a memorable, explainable visual demo.

Implementation:

1. Build the `handoff-dashboard` NitroStack widget.
2. Render the workflow as a transit line with status-coded stations.
3. Render a main-blocker hero card, health, estimated completion, evidence, fired rules, simulation comparison, planned actions, and audit log.
4. Attach the widget to blocker detection, simulation, and execution tool outputs.
5. Use the MCP host bridge for interactions when available; retain a standard tool-call path for every action.
6. Generate widget types from the final tool schemas.

Acceptance gate:

```text
The dashboard renders in NitroStudio
Laptop Allocation is visually the blocker
Evidence is readable in one interaction
Simulation displays before and after values
Execution updates the visible workflow and audit log
```

## Phase 6: Tests and Demo Rehearsal

**Goal:** Make the demo deterministic and defensible.

Implementation:

1. Add unit tests for all rules, health scoring, critical-path analysis, simulation isolation, and action validation.
2. Add integration tests for MCP resource and tool contracts.
3. Add widget-output smoke coverage where supported by the starter stack.
4. Prepare a two-minute demo script around Priya's onboarding story.
5. Test the flow through NitroStudio and one additional MCP client.

Acceptance gate:

```text
Automated tests pass
The two-minute demo can be repeated from a fresh server start
All demo claims are visible in resources, tool output, or audit history
```

## Phase 7: NitroCloud Deployment

**Goal:** Publish the MCP server with clear operational boundaries.

Implementation:

1. Add deployment configuration and an environment example without secrets.
2. Run a production build before deployment.
3. Deploy through NitroStack's NitroCloud flow.
4. Verify external MCP connectivity and resource/tool discovery.
5. Document the temporary in-memory reset behavior.
6. Record a future migration path to durable persistence.

Acceptance gate:

```text
The deployed server is reachable by an MCP client
The Priya demo completes end to end
No secret is committed
Restart behavior is documented
```

## Final Demo Sequence

1. Introduce the problem: work is stuck between teams, not inside their tools.
2. Open Priya's state and the rules through MCP.
3. Run `detect_blockers` and show Laptop Allocation as the root blocker.
4. Open evidence and fired rules.
5. Run `simulate_resolution` for Laptop Allocation today.
6. Show health rising from 62 to 86 and the critical path changing.
7. Run `plan_next_actions`, then `execute_action` with approval.
8. Show the transit stations light up and the audit log append.
9. Close with: "HandoffOS makes invisible enterprise handoffs visible, explainable, and actionable. Rules detect. AI explains. MCP acts."
